## Why

Hermes Agent (Nous Research) runs on the user's Mac mini and already exposes a rich set of channel adapters — Discord, Telegram, Slack, Matrix, BlueBubbles, Webhook, etc. — under `gateway/platforms/`. The user wants the same agent on Rokid Glasses for hands-free voice interaction, photo-grounded questions, and proactive notifications driven by Hermes scheduled jobs.

The previous draft of this proposal split the work in two: a phone-app that called `/v1/responses` directly for chat, plus a separate channel plugin used only as a webhook target for cron pushes. That split was awkward — two transports, two auth mechanisms, two code paths for what is conceptually one bidirectional conversation. Worse, it made client-driven turns and Hermes-initiated pushes diverge: they ended up sharing nothing.

This revision goes the other way: **the glasses-and-phone pair is a single Hermes channel.** We ship a Python channel adapter (`BasePlatformAdapter` subclass, idiomatic to `gateway/platforms/`) that hosts a WebSocket endpoint; the phone connects outbound and forwards everything in both directions. User text, voice, and images go up the channel; agent replies (text deltas, tool-progress, rendered TTS audio, scheduled-job pushes) come down it. There is no `/v1/responses` client on the Android side, and no phone-side STT/TTS. Inspired by `dweddepohl/clawsses` (phone-as-bridge topology, HUD UX, phone-also-shows-chat) but built fresh against Hermes' channel surface, not OpenClaw's gateway protocol. Rokid SDK details verified against `buildwithfenna/rokid-docs`.

## What Changes

- **One transport, full duplex.** Phone-app no longer calls Hermes' REST or `/v1/responses`. It holds a single outbound WebSocket to the channel adapter (`ws://mac:<port>/glasses`), authenticated by a shared secret on the upgrade request, scoped to the Tailscale interface. All chat traffic, all media, and all pushes flow over this one socket.
- **Custom Hermes channel, core (not MVP+1).** A Python module installed under `~/.hermes/plugins/hermes-channel-adapter/` registers a channel named `glasses` by subclassing `gateway/platforms/base.py:BasePlatformAdapter`. It calls `self.handle_message(...)` on inbound user turns and overrides `send(...)` / `send_image(...)` / `send_voice(...)` / `play_tts(...)` for outbound delivery. Hermes treats it identically to Discord or Matrix.
- **Voice on the server, Telegram pattern.** Long-press-to-talk records Opus *on the glasses* (the glasses have a 4-mic array, 16 kHz, accessible via standard `MediaRecorder.AudioSource.MIC` per `yodaos/docs/hardware/audio.md`). Audio bytes flow glasses → phone (via CXR `startAudioStream` / `sendStream`) → WebSocket → channel adapter → `cache_audio_from_bytes` → `event.message_type = VOICE` + `event.media_urls = [path]` → `handle_message`. The agent loop runs the STT tool itself (default: local faster-whisper, no keys). On reply, Hermes' shared TTS pipeline (`tools/tts_tool.py`, default Edge) renders Opus; our adapter overrides `send_voice` / `play_tts` to push the rendered audio frame down the WS to the phone, which forwards it to the glasses, which plays it via the local 48 kHz speaker. `/voice on` / `/voice off` commands and the auto-TTS gate at `base.py:2792-2823` work for free. **No phone-side STT/TTS; phone never touches PCM** — it relays Opus bytes between CXR and WebSocket.
- **Hermes owns sessions.** No client-side session list, no LRU workaround, no `glasses-default` bookkeeping. Sessions are managed by Hermes; the channel adapter exposes them; the glasses session picker is a thin view on top of the same WS.
- **Phone is a participating client, not a transparent bridge.** Phone-app renders chat history in a Compose UI, accepts typed input, and shows the same conversation the glasses see. Mirrors `clawsses/phone-app/.../ui/screens/MainScreen.kt:603,765-789`. Voice and camera originate from the glasses; phone is a "second screen" for the same conversation.
- **Glasses-app uses CXR-S *and* CXR-L.** Previous draft only mentioned CXR-S. CXR-L is the documented surface for `takePhoto(width,height,quality)`, `startAudioStream(codecType)`, and structured HUD push (`openCustomView(json)`); we use CXR-L for camera and audio I/O and CXR-S for the bidirectional message wire to the phone. CXR-L requires `com.rokid.sprite.aiapp` (versionCode ≥ 100000) installed on the device.
- **CXR `sendCustomCmd` chunking is dropped.** Clawsses' 500-char truncation is a clawsses-imposed convention, not a Rokid limit. Caps is binary and supports nested `byte[]`; bulk transport is `sendStream(CxrStreamType, byte[], fileName, cb)`. We use Caps directly for control frames and `sendStream` for audio/image bulk. (Confirmed against `cxr-m/sdk-decompiled-reference.md`.)
- **Touchpad gestures consume standard Android `KeyEvent`s** (DPAD_LEFT/RIGHT/UP/DOWN/CENTER, BACK, KEYCODE_CAMERA, etc.). The Rokid touchpad is a capacitive sensor on the right temple surfaced as Android key events — there is no documented gesture taxonomy beyond key codes, so we don't write one.
- **APK sideload is documented.** `CxrApi.startUploadApk(wifiAddress, callback)` (`cxr-m/sdk-decompiled-reference.md:1007-1017`) does multipart POST of the bundled glasses APK to `http://<glasses_ip>:8848`, with `ApkStatusCallback` reporting upload/install/uninstall/openApp. Same pattern clawsses uses, but officially on-spec.

## Capabilities

### New Capabilities

- `glasses-hud`: HUD app on Rokid Glasses — Compose UI on 480×640 portrait, key-event gesture routing, on-glasses camera capture (≤256 KB JPEG via CXR-L `takePhoto`), on-glasses audio recording (Opus via standard Android `MediaRecorder`) and playback (via `MediaPlayer` on the glasses' 48 kHz speaker), session picker, wake-on-message, streaming text-chunk rendering, tool-progress subline. Bluetooth wire to phone via Rokid CXR-S; media APIs via CXR-L.
- `phone-bridge`: Android companion app — holds the outbound WebSocket to the Hermes channel adapter, runs the Bluetooth/CXR-M bridge to the glasses, relays Opus audio frames in both directions (no PCM handling on the phone), coordinates wake-on-message, sideloads the glasses APK on first launch via `startUploadApk`, **renders chat history with a typed-input affordance**, exposes a settings screen.
- `hermes-client-protocol`: Shared Kotlin module — DTOs and parsers for the phone↔channel WebSocket frame protocol AND the phone↔glasses Bluetooth envelope protocol (over CXR Caps). Single `parseFrame(json)` entry point with a `type` discriminator. No Hermes REST DTOs.
- `glasses-channel-plugin`: Python Hermes channel adapter — subclasses `BasePlatformAdapter`, exposes `glasses` as a delivery target, hosts the WebSocket endpoint, dispatches inbound text and voice/image frames as `handle_message(event)` calls, pushes Hermes-originated content (assistant text deltas, rendered TTS audio, tool-progress, cron / run-completion / agent-nudge messages with origin metadata) as outbound frames. Configurable via `~/.hermes/plugins/hermes-channel-adapter/config.yaml`.

### Modified Capabilities

None. Greenfield project.

## Impact

- **New code:** Three Android Gradle modules (Kotlin + Jetpack Compose) plus one Python channel-adapter module. No existing code to migrate.
- **External dependencies (Android):** Rokid CXR-M (phone), CXR-S + CXR-L (glasses) from `https://maven.rokid.com/repository/maven-public/`; OkHttp (HTTP + WebSocket); Gson. No `SpeechRecognizer`, no Edge TTS client, no ElevenLabs SDK on the phone.
- **External dependencies (Python):** `aiohttp` (already used by `gateway/platforms/webhook.py` and `api_server.py`) for the WebSocket server; whatever Hermes brings for `BasePlatformAdapter` and the shared voice pipeline.
- **Hermes config:** Adapter installed under `~/.hermes/plugins/hermes-channel-adapter/`; `config.yaml` provides `listen_host`, `listen_port`, `shared_secret`. Hermes gateway must be restarted to register the channel. Voice pipeline already configured at the Hermes level (`stt.provider`, `tts.provider`, `voice.auto_tts`) — adapter inherits it. Tailscale up on both ends.
- **Glasses prerequisites:** `com.rokid.sprite.aiapp` (versionCode ≥ 100000) must be installed for CXR-L `AuthorizationHelper`. Min SDK 28, target SDK ≤ 32, arm64-v8a.
- **User credentials:** `local.properties` for Rokid SDK credentials; phone-app settings UI for Hermes WebSocket URL and shared secret. Shared secret stored in Android EncryptedSharedPreferences.
- **Out of scope:** No iOS, no other AR vendors, no public-internet exposure (Tailscale-only), no phone-local STT/TTS, no Discord/Telegram/Slack mirroring on glasses (those are independent Hermes channels), no fork or vendoring of clawsses code, no eye/SLAM tracking (not exposed by Rokid SDK).
